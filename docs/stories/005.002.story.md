## Overview

Implement Stripe SetupIntent API for secure payment method tokenization enabling patients to save cards for future use. Implements PCI DSS SAQ A compliance with RLS policies preventing cross-patient access and LGPD soft-delete strategy for patient data.

---

## User Story Statement

**As a** patient
**I want** to save my payment method securely for future transactions
**So that** I can quickly pay for appointments without re-entering card details

---

## Acceptance Criteria

**SetupIntent Flow:**
- [ ] POST `/api/patient/{id}/payment-methods/create-setup-intent` creates SetupIntent
- [ ] SetupIntent returns setup_intent_id + client_secret
- [ ] POST `/api/patient/{id}/payment-methods` confirms SetupIntent
- [ ] Payment method saved in payment_methods table
- [ ] Only payment_method_id (tokenized) stored (no raw card data)

**Payment Method Operations:**
- [ ] GET `/api/patient/{id}/payment-methods` lists saved methods
- [ ] Response includes: payment_method_id, brand, last4, exp_month, exp_year, is_default
- [ ] DELETE `/api/patient/{id}/payment-methods/{method_id}` removes method
- [ ] Soft-delete: Set deleted_at timestamp
- [ ] Cascade delete: If patient deleted, all payment methods deleted

**RLS Security:**
- [ ] RLS policy: Patients see only own payment methods
- [ ] RLS policy: Clinic staff cannot see patient payment methods
- [ ] Cross-patient access returns 403 Forbidden
- [ ] Audit log entry: operation: method_saved or method_deleted

**Card Metadata:**
- [ ] Card brand stored (visa, mastercard, elo, amex)
- [ ] Card last4 stored (no full PAN)
- [ ] Exp month/year stored
- [ ] is_default flag for default payment method
- [ ] created_at timestamp

**LGPD Compliance:**
- [ ] Soft-delete strategy (deleted_at column)
- [ ] No PII in payment_methods table (only payment_method_id)
- [ ] Audit log captured for all operations
- [ ] 5-year retention for audit logs

**Database:**
- [ ] payment_methods table created
- [ ] Indexes on patient_id, clinic_id
- [ ] RLS policies enabled
- [ ] Cascade delete policies

**Testing:**
- [ ] Unit tests for SetupIntent (≥70% coverage)
- [ ] RLS tests (cross-patient access blocked)
- [ ] Cascade delete tests
- [ ] LGPD soft-delete verified

## Test Scenarios

**2.1: Save Payment Method**
- SetupIntent created, patient enters card, confirmed
- Payment method saved in database
- Audit log: method_saved, success: true

**2.2: List Payment Methods**
- GET endpoint returns saved methods
- Only patient's own methods listed
- Cross-patient access blocked (403)

**2.3: Delete Payment Method**
- DELETE endpoint removes method
- deleted_at set, not physically deleted
- Audit log: method_deleted

**2.4: RLS Cross-Patient Access**
- Patient A tries to access Patient B's methods
- Returns 403 Forbidden
- Audit log: RLS violation attempt

**2.5: Cascade Delete on Patient Deletion**
- Patient soft-deleted
- All payment methods also soft-deleted
- Stripe payment_method_id NOT revoked (separate)
