# User Story: Stripe PaymentIntent API with 3D Secure Support

**Epic:** EPIC-005 — Stripe Payment Integration
**Title:** Create Stripe PaymentIntent API endpoint with 3D Secure support
**ID:** 005.001
**Created:** 2026-05-17
**Status:** Draft
**Ready for:** @po validation

## Overview

Implement Stripe PaymentIntent API endpoint to process appointment payments securely via Stripe Elements with PCI DSS compliance (SAQ A). Support 3D Secure (SCA) authentication, webhook event processing, and full audit logging.

---

## User Story Statement

**As a** clinic staff member or patient
**I want** to process secure payments for clinic appointments via Stripe
**So that** appointments can be paid for online with PCI-compliant card tokenization and 3D Secure support

---

## Acceptance Criteria

**Core Functionality:**
- [ ] POST `/api/payments/process` endpoint created
- [ ] Stripe PaymentIntent created with metadata
- [ ] Card tokenization via Stripe.js (no raw card data)
- [ ] Payment status saved to appointments table
- [ ] Webhook endpoint `/api/webhooks/stripe` receives events
- [ ] Webhook signature validation via stripe.webhooks.constructEvent()
- [ ] Idempotency implemented (Idempotency-Key header)

**3D Secure Support:**
- [ ] POST `/api/payments/{id}/confirm` for 3D Secure confirmation
- [ ] Client secret returned when requires_action
- [ ] Redirect URL provided for 3D Secure challenge

**Error Handling:**
- [ ] Errors: card_declined, processing_error, rate_limit_error, expired_card, incorrect_cvc
- [ ] PT-BR error messages (no internal details)
- [ ] Retry logic (exponential backoff: 1s, 5s, 30s)
- [ ] Max 3 retries for transient errors

**Audit Integration:**
- [ ] Appointment status = "paid" on success
- [ ] Audit log entry created
- [ ] No card data in logs (only payment_method_id)

**Database:**
- [ ] payments table created
- [ ] webhook_events table created
- [ ] appointments table updated

**Testing:**
- [ ] Unit tests (≥70% coverage)
- [ ] Integration tests for webhooks
- [ ] Error scenarios tested
- [ ] Webhook idempotency verified

## Test Scenarios

**1.1: Happy Path**
- Valid card (4242): Payment succeeds, appointment paid, audit log created

**1.2: Card Declined**
- Declined card (4000 0000 0000 0002): Returns error, appointment unchanged

**1.3: 3D Secure**
- 3DS card (4000 0000 0000 0127): Requires action, confirm completes payment

**1.4: Idempotency**
- Duplicate request with same key: Returns cached response

**1.5: Webhook Idempotency**
- Duplicate webhook: Processed once only

**1.6: Webhook Timeout**
- Long processing: Returns 200 OK within 30s, completes async
