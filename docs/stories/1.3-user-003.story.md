## Overview

Implement role-based access control system with roles table, user-role assignment, and Row-Level Security (RLS) policies to enforce clinic-level data isolation and role-based permissions.

---

## User Story Statement

**As a** clinic admin
**I want** to assign roles (Admin, Doctor, Receptionist, Patient) to users
**So that** users only access features and data appropriate to their role

---

## Acceptance Criteria

- [x] `roles` table created with pre-populated roles (admin, doctor, receptionist, patient)
- [x] `user_roles` junction table links users to roles
- [x] RLS policies restrict data access by clinic_id and user role
- [x] Admin users can view/edit all clinic data
- [x] Doctor users can view/edit patient data and appointments
- [x] Receptionist users can view/edit patients and appointments (no sensitive data)
- [x] Patient users can view only their own data

## Technical Requirements

**Database Schema:**
```sql
CREATE TABLE public.roles (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT UNIQUE NOT NULL,
  description TEXT,
  permissions JSONB DEFAULT '[]',
  created_at TIMESTAMP DEFAULT now()
);

CREATE TABLE public.user_roles (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  role_id UUID NOT NULL REFERENCES roles(id) ON DELETE CASCADE,
  clinic_id UUID NOT NULL,
  assigned_at TIMESTAMP DEFAULT now(),
  UNIQUE(user_id, role_id, clinic_id)
);

-- Pre-populated roles:
INSERT INTO roles (name, description) VALUES
  ('admin', 'Full system access'),
  ('doctor', 'Medical professional - full clinical access'),
  ('receptionist', 'Receptionist - scheduling & patient intake'),
  ('patient', 'Patient - self-service portal');
```

**RLS Policies:**
- Clinic-level isolation: Users only see data for their clinic
- Role-based access: Policies enforce read/write based on user role
- Patient self-access: Patients can view only their own records

**Indexes:**
- `idx_user_roles_user_id` on user_roles(user_id)
- `idx_user_roles_clinic_id` on user_roles(clinic_id)
- `idx_user_roles_role_id` on user_roles(role_id)

**Frontend:**
- Role checks in middleware/guards (redirect if insufficient permission)
- Display UI elements based on user role (e.g., "Admin" tab only for admins)
- Load user role on app initialization

**Backend:**
- Helper function: `getUserRole(user_id, clinic_id)` returns user's role
- Helper function: `hasPermission(user_id, action, resource)` for fine-grained control
- RLS policies enforced at database level (no bypass possible)

**Security:**
- RLS policies prevent SQL injection (parameterized queries)
- Role-based access enforced at database, not just frontend
- No privilege escalation: Users cannot modify their own role

**Testing:**
- Unit test: Role assignment, permission checking logic
- Integration test: RLS policies enforce access correctly
- Security test: Attempt unauthorized access, verify RLS blocks it
