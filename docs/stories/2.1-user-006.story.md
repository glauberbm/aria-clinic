# User Story: USER-006

**Epic:** EPIC-002 — Patient Management & Health Records
**Title:** Patient Registration & Profile Setup
**ID:** USER-006
**Created:** 2026-05-15
**Status:** Ready for Review

---

## Overview

Implement patient self-registration flow, email verification, and initial profile creation. Establishes RLS policies for patient data isolation and integration with clinic admin dashboard for registration approvals.

---

## User Story Statement

**As a** patient
**I want** to register for an account and set up my profile
**So that** I can access clinic services and manage my health information

---

## Acceptance Criteria

- [ ] Patient registration form at `/auth/register` with email, password, name, birth date, phone
- [ ] Email verification link sent to patient after registration (Supabase auth + custom email trigger)
- [ ] Patient profile auto-created in `public.patients` table after email verification
- [ ] RLS policies enforce: patients see only their own records + clinic staff see only their clinic's patients
- [ ] Admin dashboard updates to show pending registrations for approval
- [ ] Verification timeout: Links expire after 24 hours
- [ ] Error handling: Duplicate email, invalid format, network errors with user-friendly messages

---

## Technical Requirements

**Frontend:**
- Create `/auth/register` page with form validation (Zod schema)
- Email and password validation (5+ chars, strong password rules)
- Conditional rendering: Show verification email confirmation message
- Link to login page after successful registration
- Responsive design (mobile-first)

**Backend:**
- POST `/api/auth/register` — Create Supabase auth user + send verification email
- Email trigger function in Supabase: On auth.users INSERT, send custom email via SendGrid/Resend
- Verification webhook: On email link click, auto-create patient profile in `public.patients`
- POST `/api/auth/verify` — Verify email token and finalize registration
- Error handling: Standardized error responses (400, 409 for duplicate, 500 for server errors)

**Database:**
- Create `public.patients` table with RLS policies
- Email verification required
- Create `public.pending_registrations` table for admin workflow

**Security:**
- Email verification required (no access until verified)
- Password strength: 5+ chars, min 1 uppercase, 1 number, 1 special char
- Rate limit: 5 registration attempts per IP per hour

**Testing:**
- Unit test: Email validation, password validation
- Integration test: Registration flow end-to-end
- Security test: RLS validation, unauthorized access blocked
